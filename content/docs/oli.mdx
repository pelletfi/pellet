**OLI** is Pellet's Open-Ledger Interface — a live observability layer over autonomous economic activity on Tempo. Every MPP transfer the chain emits is ingested, decoded, and surfaced with provenance. Every Pellet Wallet payment lands here automatically.

The dashboard is at [pellet.network/oli](/oli). The methodology is at [/oli/methodology](/oli/methodology).

## What OLI tracks

- **Transfer events** on watched TIP-20 stablecoins (USDC.e, USDT0). Hourly cron ingests the chain.
- **Watched entities (`agents`)** — curated MPP services and infrastructure addresses. Currently 7 rows: Pellet, Stargate USDC bridge, Tether USDT0 mint, TIP-20 factory, Enshrined DEX precompile, Tempo MPP Gateway, Codex MPP service.
- **Per-event provenance** — each `agent_events` row carries a methodology version + source block + tx hash. Re-deriving any number is a matter of replaying the matcher against the raw events table.
- **Gateway routing attribution** — when a payment routes through the Tempo MPP Gateway aggregator (`0xca4e835f…`), OLI recovers the underlying provider via two paths:
    - **Pattern A** — the gateway's escrow contract emits a `Settlement` event whose topic[2] is the provider address. Recovers ~9% of gateway txs as named addresses.
    - **Pattern B** — user→gateway calldata carries a `bytes32 ref` whose bytes 5–14 are a stable per-service fingerprint set by Tempo's MPP client. Groups the remaining ~91% of gateway txs by service even when the provider address can't be recovered.
- **Live SSE feed** — `/api/oli/feed` pushes new agent_events as they're matched. Off by default in production (cost) — re-enable via `NEXT_PUBLIC_OLI_LIVE_FEED=1`.

For the full pipeline (event ingestion → matching → attribution → provenance), read [Methodology](/docs/methodology).

## Methodology in one screen

```
events (raw chain logs, idempotent by tx_hash + log_index)
  ↓ matcher (lib/ingest/matcher.ts) — methodology v0.2
agent_events (decoded, joined to watched agents)
  ↓ /api/cron/attribute (every 6h)
  ↓ Pattern A: Settlement event from 0x33b9…4f25 → routed_to_address
  ↓ Pattern B: calldata bytes 5–14   → routed_fingerprint
agent_events (now with attribution)
  ↓ LEFT JOIN address_labels
recentDecoded / serviceDetail / providerDetail / etc.
```

Every row carries `methodology_version` (currently `v0.2`) and `source_block`. Anyone can re-derive any number with the raw `events` table + the matcher source.

## Querying OLI

OLI exposes JSON endpoints under `/api/oli/`. See [API reference](/docs/api) for the full list.

```bash
# Headline stats
curl https://pellet.network/api/oli/dashboard?w=24h

# Top services
curl https://pellet.network/api/oli/services

# Single event
curl https://pellet.network/api/oli/events/1234

# Search (events by tx hash, agents/services by label, addresses)
curl 'https://pellet.network/api/oli/search?q=stargate'
```

For agent installation: [`pellet.network/skill.md`](/skill.md). Read it to teach an agent when to use OLI.
