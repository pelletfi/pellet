# Pellet Wallet — Research Synthesis (2026-04-29)

Four parallel research agents on: Stripe Link Wallet, WebAuthn PRF self-custody patterns, Tempo's chain primitives, and the broader agent-payment competitive landscape. This doc captures the architecture-shifting findings + the v0 plan they imply.

## 1. The architecture changed

Pre-research, we had two viable paths:

- **B1 / B2** — passkey-rooted self-custody via WebAuthn PRF, deriving a secp256k1 key on-device per signature
- **C** — speculative "Tempo-native AA delegation, if it exists"

**Path C exists, is shipped, and is structurally cleaner than B.** Tempo has a first-class delegation precompile and a transaction envelope that natively supports passkey signatures. We should build on it directly instead of deriving keys via PRF.

### Tempo primitives we should treat as canonical

| Primitive | Where | What it gives us |
|---|---|---|
| **TempoTransaction (type `0x76`)** | EIP-2718 envelope on Tempo | Native AA: `calls[]` batch, `feePayerSignature` for sponsored gas, `validBefore`/`validAfter` for tx-level expiration, `aa_authorization_list` (EIP-7702-style), `key_authorization` to provision keys atomically. Sender signature accepts secp256k1, P256, **and WebAuthn**. |
| **AccountKeychain precompile** | `0xAAAAAAAA00000000000000000000000000000000` | On-chain session keys. `authorizeKey(keyId, sigType, expiry, enforceLimits, limits[])` — `sigType` 0/1/2 = secp/P256/WebAuthn. Caps tracked per-token at protocol level. T3 upgrade adds `period` (recurring), `CallScope[]` (per-target/selector). |
| **TIP-20 stables (USDC.e, USDT0)** | `0x20c0…b9537d11c60e8b50` / `0x20c0…14f22ca97301eb73` | Native stable contracts. `transferWithMemo(address, uint256, bytes32)` selector `0x95777d59` is the canonical x402 settlement call. **Do not reach for EIP-3009 — these don't implement it.** They support `permit` and `transferWithMemo`. |
| **TempoStreamChannel** | `0x33b9…4f25` | Payment-channel / EIP-712 voucher escrow. **Not** the per-request charge target — only used for `intent="session"` 402 challenges. `intent="charge"` settles via direct `transferWithMemo` to `recipient`. |

### What this means for Pellet Wallet

- **No WebAuthn PRF.** No HKDF→secp256k1 derivation. No 32-byte-mod-n footgun. Tempo verifies WebAuthn signatures natively — passkey signs the `TempoTransaction` envelope directly.
- **No smart-contract wallet.** No ERC-4337, no kernel module setup. AccountKeychain is the policy engine, enshrined in the protocol.
- **No session key encryption.** Caps + expiry live on-chain; revocation is a one-tx `revokeKey` call.
- **No EIP-3009.** TIP-20 `transferWithMemo` is the canonical x402 call on Tempo.

The new flow:

1. User has a Tempo account rooted in a passkey (WebAuthn `sigType=2`)
2. User calls `AccountKeychain.authorizeKey(agentKey, sigType=0, expiry, true, [{USDC.e, 10_000000}])` — grants the agent key $10 cap on USDC.e
3. Agent receives 402 challenge, builds `TempoTransaction { calls: [{ to: USDC.e, input: transferWithMemo(recipient, amount, challengeId) }] }`, signs with agent key
4. Tempo enforces the cap at execution; tx confirms or reverts
5. Tx hash flows into OLI via existing ingestion → user sees their wallet's receipts on the public dashboard automatically

## 2. The Stripe Link CLI is the template

Stripe shipped `link-cli` (github.com/stripe/link-cli) with this exact onboarding:

```
> install link-cli                       (CLI install)
> link auth-start                        (device-code pairing)
> Visit link.com/device?code=apple-grape-orange
> When prompted, verify the passphrase: apple-grape-orange
> Waiting for approval…                  (browser approval flow)
```

User goes to `link.com/device`, approves on phone, CLI gets a session token. **Every subsequent spend blocks on a fresh mobile approval.** No spend-cap delegation, no recurring, no streaming. Listed as "Coming soon: Granular agent controls."

That's the gap.

### What we mirror

- Tool naming parity (`auth start`, `auth status`, `pay`, `revoke`) so agents trained on Stripe's CLI work without retraining
- Device-code pairing flow (CLI prints code + URL, user approves in browser, CLI polls for token)
- skill.md as the install convention (we already ship `pellet.network/skill.md`)
- Drop-in MCP server bundled with the CLI for one-line install in Claude Code / Cursor / Cloudflare Agents

### What we diverge on (the wedge)

- **Spend-cap delegation as v1, not roadmap.** AccountKeychain authorize-once-with-caps means the agent is autonomous within the caps. No per-call mobile prompt. This is the demo.
- **Public on-chain receipts.** Every payment is a deep-linkable `pellet.network/oli/event/[id]` URL. Stripe's are private Stripe IDs.
- **Two-mode 402 evaluator.** When a 402 advertises both `method="tempo"` and `method="stripe"`, our CLI tells the agent we're the right rail; when only `stripe`, we acknowledge we can't fulfill. Honest interop, not fake competition.
- **Open source, single binary.** `@pelletnetwork/cli` works for any agent runtime. No Stripe-side gating.

## 3. Competitive map (compressed)

| Player | Pattern | Our position vs them |
|---|---|---|
| **Stripe Link Wallet** | Custodial, card-rail, per-spend approval, owns Privy + Tempo | Open self-custody on their own chain. Compatible (skill.md), differentiated (caps, on-chain receipts) |
| **Coinbase Agentic Wallets + x402 Bazaar** | TEE-secured wallets, Base-first, 480k agents, $50M throughput | Tempo-native (vs Base), passkey-rooted (vs TEE-rooted), auto-OLI receipts |
| **Crossmint Lobster Cash** | Self-custody smart wallets, dual stable+Visa rail | Tempo-only, simpler primitive, public receipts |
| **Privy / Turnkey** | Embedded wallet infra (TEE+SSS) — the "Stripe stack" | We're not embedded — we're an open agent CLI. Different distribution layer. |
| **Skyfire / Halliday / Nevermined / Payman** | Various custodial agent-money rails | Open self-custody is the consistent differentiator |

The unscarce position: **open self-custody, USDC-native, passkey-rooted, on Tempo, with auto-OLI receipts**. Two of those four are common — passkey-root + auto-OLI is the differentiation. Tempo as home chain is also distinctive (Coinbase=Base, Crossmint=EVM/Sol).

## 4. v0 plan — what we ship next

In strict order. Each step is independently shippable + reviewable.

### Step 1 — CLI scaffold (new repo: `pelletnetwork/cli`)
- `npx @pelletnetwork/cli` entrypoint
- `pellet auth start` — generates device code, opens browser, polls for token
- `pellet auth status` — shows active session + caps
- `pellet auth revoke` — local-only revoke (server revoke comes with backend)
- No payment logic yet. Validates the install + pairing UX end-to-end.

### Step 2 — Device-code endpoints on `pellet.network`
- `POST /api/wallet/device/start` — issues `(code, deviceId)` pair, stores pending session
- `GET  /api/wallet/device/poll`  — CLI polls until user approves; returns bearer token on success
- `GET  /wallet/device?code=…`    — browser approval page; passkey enrollment if first-time

### Step 3 — Passkey enrollment + AccountKeychain authorize
- WebAuthn `navigator.credentials.create()` on first device pairing
- After approval, browser submits `authorizeKey` tx to Tempo with the chosen caps
- Server records the resulting `keyId` against the device session

### Step 4 — `pellet pay` (the demo moment)
- Agent calls `pellet pay <402-url>` (or programmatic API equivalent)
- CLI builds TempoTransaction, signs with agent key, submits via sponsored relayer
- Returns the tx hash + the OLI event URL (which lights up after the next ingest cycle)
- Same demo Stripe shows in their video — but autonomous, no per-call mobile prompt

### Step 5 — MCP server bundle
- `@pelletnetwork/cli` exports an MCP server with the same tools
- Drop into Claude Code / Cursor / Cloudflare Agents in one config line
- Distribution play: Anthropic's MCP-as-distribution surface is the live wedge per the landscape research

### Step 6 — Demo video + launch
- Same shape as Stripe's 28-second video
- Tweet from @pelletnetwork: "Stripe shipped Link Wallet for Agents. We shipped the open self-custody version on Tempo. No mobile approval per spend — your agent has caps, not a leash."
